31  <h2><a name="overview">Overview</a></h2>
32  Coprocessors are code that runs in-process on each region server. Regions
33  contain references to the coprocessor implementation classes associated
34  with them. Coprocessor classes can be loaded either from local
35  jars on the region server's classpath or via the HDFS classloader.
36  <p>
37  Multiple types of coprocessors are provided to provide sufficient flexibility
38  for potential use cases. Right now there are:
39  <p>
40  <ul>
41  <li>Coprocessor: provides region lifecycle management hooks, e.g., region
42  open/close/split/flush/compact operations.</li>
43  <li>RegionObserver: provides hook for monitor table operations from
44  client side, such as table get/put/scan/delete, etc.</li>
45  <li>Endpoint: provides on demand triggers for any arbitrary function
46  executed at a region. One use case is column aggregation at region
47  server. </li>
48  </ul>
49  
50  <h2><a name="coprocessor">Coprocessor</a></h2>
51  A coprocessor is required to
52  implement <code>Coprocessor</code> interface so that coprocessor framework
53  can manage it internally.
54  <p>
55  Another design goal of this interface is to provide simple features for
56  making coprocessors useful, while exposing no more internal state or
57  control actions of the region server than necessary and not exposing them
58  directly.
59  <p>
60  Over the lifecycle of a region, the methods of this interface are invoked
61  when the corresponding events happen. The master transitions regions
62  through the following states:
63  <p>
64  &nbsp;&nbsp;&nbsp;
65  unassigned -> pendingOpen -> open -> pendingClose -> closed.
66  <p>
67  Coprocessors have opportunity to intercept and handle events in
68  pendingOpen, open, and pendingClose states.
69  <p>
70  
71  <h3>PendingOpen</h3>
72  <p>
73  The region server is opening a region to bring it online. Coprocessors
74  can piggyback or fail this process.
75  <p>
76  <ul>
77    <li>preOpen, postOpen: Called before and after the region is reported as
78   online to the master.</li><p>
79  </ul>
80  <p>
81  <h3>Open</h3>
82  The region is open on the region server and is processing both client
83  requests (get, put, scan, etc.) and administrative actions (flush, compact,
84  split, etc.). Coprocessors can piggyback administrative actions via:
85  <p>
86  <ul>
87    <li>preFlush, postFlush: Called before and after the memstore is flushed
88    into a new store file.</li><p>
89    <li>preCompact, postCompact: Called before and after compaction.</li><p>
90    <li>preSplit, postSplit: Called after the region is split.</li><p>
91  </ul>
92  <p>
93  <h3>PendingClose</h3>
94  The region server is closing the region. This can happen as part of normal
95  operations or may happen when the region server is aborting due to fatal
96  conditions such as OOME, health check failure, or fatal filesystem
97  problems. Coprocessors can piggyback this event. If the server is aborting
98  an indication to this effect will be passed as an argument.
99  <p>
100 <ul>
101   <li>preClose and postClose: Called before and after the region is
102   reported as closed to the master.</li><p>
103 </ul>
104 <p>
105 
106 <h2><a name="regionobserver">RegionObserver</a></h2>
107 If the coprocessor implements the <code>RegionObserver</code> interface it can
108 observe and mediate client actions on the region:
109 <p>
110 <ul>
111   <li>preGet, postGet: Called before and after a client makes a Get
112   request.</li><p>
113   <li>preExists, postExists: Called before and after the client tests
114   for existence using a Get.</li><p>
115   <li>prePut and postPut: Called before and after the client stores a value.
116   </li><p>
117   <li>preDelete and postDelete: Called before and after the client
118   deletes a value.</li><p>
119   <li>preScannerOpen postScannerOpen: Called before and after the client
120   opens a new scanner.</li><p>
121   <li>preScannerNext, postScannerNext: Called before and after the client
122   asks for the next row on a scanner.</li><p>
123   <li>preScannerClose, postScannerClose: Called before and after the client
124   closes a scanner.</li><p>
125   <li>preCheckAndPut, postCheckAndPut: Called before and after the client
126   calls checkAndPut().</li><p>
127   <li>preCheckAndDelete, postCheckAndDelete: Called before and after the client
128   calls checkAndDelete().</li><p>
129 </ul>
130 You can also extend abstract class <code>BaseRegionObserverCoprocessor</code>
131 which
132 implements both <code>Coprocessor</code> and <code>RegionObserver</code>.
133 In addition, it overrides all methods with default behaviors so you don't
134 have to override all of them.
135 <p>
136 Here's an example of what a simple RegionObserver might look like. This
137 example shows how to implement access control for HBase. This
138 coprocessor checks user information for a given client request, e.g.,
139 Get/Put/Delete/Scan by injecting code at certain
140 <code>RegionObserver</code>
141 preXXX hooks. If the user is not allowed to access the resource, a
142 CoprocessorException will be thrown. And the client request will be
143 denied by receiving this exception.
144 <div style="background-color: #cccccc; padding: 2px">
145 <blockquote><pre>
146 package org.apache.hadoop.hbase.coprocessor;
147 
148 import java.util.List;
149 import org.apache.hadoop.hbase.KeyValue;
150 import org.apache.hadoop.hbase.client.Get;
151 
152 // Sample access-control coprocessor. It utilizes RegionObserver
153 // and intercept preXXX() method to check user privilege for the given table
154 // and column family.
155 public class AccessControlCoprocessor extends BaseRegionObserverCoprocessor {
156   // @Override
157   public Get preGet(CoprocessorEnvironment e, Get get)
158       throws CoprocessorException {
159 
160     // check permissions..
161     if (access_not_allowed)  {
162       throw new AccessDeniedException(&quot;User is not allowed to access.&quot;);
163     }
164     return get;
165   }
166 
167   // override prePut(), preDelete(), etc.
168 }
169 </pre></blockquote>
170 </div>

172 <h2><a name="commandtarget">Endpoint</a></h2>
173 <code>Coprocessor</code> and <code>RegionObserver</code> provide certain hooks
174 for injecting user code running at each region. The user code will be triggered
175 by existing <code>HTable</code> and <code>HBaseAdmin</code> operations at
176 the certain hook points.
177 <p>
178 Coprocessor Endpoints allow you to define your own dynamic RPC protocol to communicate
179 between clients and region servers, i.e., you can create a new method, specifying custom
180 request parameters and return types.  RPC methods exposed by coprocessor Endpoints can be
181 triggered by calling client side dynamic RPC functions -- <code>HTable.coprocessorService(...)
182 </code>.
183 <p>
184 To implement an Endpoint, you need to:
185 <ul>
186  <li>Define a protocol buffer Service and supporting Message types for the RPC methods.
187  See the
188  <a href="https://developers.google.com/protocol-buffers/docs/proto#services">protocol buffer guide</a>
189  for more details on defining services.</li>
190  <li>Generate the Service and Message code using the protoc compiler</li>
191  <li>Implement the generated Service interface in your coprocessor class and implement the
192  <code>CoprocessorService</code> interface.  The <code>CoprocessorService.getService()</code>
193  method should return a reference to the Endpoint's protocol buffer Service instance.
194 </ul>
195 <p>
196 For a more detailed discussion of how to implement a coprocessor Endpoint, along with some sample
197 code, see the {@link org.apache.hadoop.hbase.client.coprocessor} package documentation.
198 </p>

200 <h2><a name="load">Coprocessor loading</a></h2>
201 A customized coprocessor can be loaded by two different ways, by configuration,
202 or by <code>HTableDescriptor</code> for a newly created table.
203 <p>
204 (Currently we don't really have an on demand coprocessor loading mechanism for
205 opened regions.)
206 <h3>Load from configuration</h3>
207 Whenever a region is opened, it will read coprocessor class names from
208 <code>hbase.coprocessor.region.classes</code> from <code>Configuration</code>.
209 Coprocessor framework will automatically load the configured classes as
210 default coprocessors. The classes must be included in the classpath already.
211 
212 <p>
213 <div style="background-color: #cccccc; padding: 2px">
214 <blockquote><pre>
215   &lt;property&gt;
216     &lt;name&gt;hbase.coprocessor.region.classes&lt;/name&gt;
217     &lt;value&gt;org.apache.hadoop.hbase.coprocessor.AccessControlCoprocessor, org.apache.hadoop.hbase.coprocessor.ColumnAggregationProtocol&lt;/value&gt;
218     &lt;description&gt;A comma-separated list of Coprocessors that are loaded by
219     default. For any override coprocessor method from RegionObservor or
220     Coprocessor, these classes' implementation will be called
221     in order. After implement your own
222     Coprocessor, just put it in HBase's classpath and add the fully
223     qualified class name here.
224     &lt;/description&gt;
225   &lt;/property&gt;
226 </pre></blockquote>
227 </div>
228 <p>
229 The first defined coprocessor will be assigned
230 <code>Coprocessor.Priority.SYSTEM</code> as priority. And each following
231 coprocessor's priority will be incremented by one. Coprocessors are executed
232 in order according to the natural ordering of the int.
233 
234 <h3>Load from table attribute</h3>
235 Coprocessor classes can also be configured at table attribute. The
236 attribute key must start with "Coprocessor" and values of the form is
237 &lt;path&gt;:&lt;class&gt;:&lt;priority&gt;, so that the framework can
238 recognize and load it.
239 <p>
240 <div style="background-color: #cccccc; padding: 2px">
241 <blockquote><pre>
242 &#39;COPROCESSOR$1&#39; =&gt; &#39;hdfs://localhost:8020/hbase/coprocessors/test.jar:Test:1000&#39;
243 &#39;COPROCESSOR$2&#39; =&gt; &#39;/hbase/coprocessors/test2.jar:AnotherTest:1001&#39;
244 </pre></blockquote>
245 </div>
246 <p>
247 &lt;path&gt; must point to a jar, can be on any filesystem supported by the
248 Hadoop </code>FileSystem</code> object.
249 <p>
250 &lt;class&gt; is the coprocessor implementation class. A jar can contain
251 more than one coprocessor implementation, but only one can be specified
252 at a time in each table attribute.
253 <p>
254 &lt;priority&gt; is an integer. Coprocessors are executed in order according
255 to the natural ordering of the int. Coprocessors can optionally abort
256 actions. So typically one would want to put authoritative CPs (security
257 policy implementations, perhaps) ahead of observers.
258 <p>
259 <div style="background-color: #cccccc; padding: 2px">
260 <blockquote><pre>
261   Path path = new Path(fs.getUri() + Path.SEPARATOR +
262     "TestClassloading.jar");
263 
264   // create a table that references the jar
265   HTableDescriptor htd = new HTableDescriptor(TableName.valueOf(getClass().getTableName()));
266   htd.addFamily(new HColumnDescriptor("test"));
267   htd.setValue("Coprocessor$1",
268     path.toString() +
269     ":" + classFullName +
270     ":" + Coprocessor.Priority.USER);
271   HBaseAdmin admin = new HBaseAdmin(this.conf);
272   admin.createTable(htd);
273 
274 <h3>Chain of RegionObservers</h3>
275 As described above, multiple coprocessors can be loaded at one region at the
276 same time. In case of RegionObserver, you can have more than one
277 RegionObservers register to one same hook point, i.e, preGet(), etc.
278 When a region reach the
279 hook point, the framework will invoke each registered RegionObserver by the
280 order of assigned priority.
281 
282 </pre></blockquote>
283 </div>
